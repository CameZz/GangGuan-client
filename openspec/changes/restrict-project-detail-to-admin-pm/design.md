## Context

项目详情页已有路由 `/project/:id` 和页面组件 `ProjectDetailView.vue`，近期还承载了项目阶段库等管理功能。当前顶部导航只有看板、列表、时间轴、成员排期、成员等入口，没有明确的项目详情入口；路由只检查登录和项目选择，不区分 PM 与普通成员。

## Goals / Non-Goals

**Goals:**
- 为管理员和 PM 在当前项目导航中提供“项目详情”入口。
- 普通成员不看到项目详情入口。
- 普通成员直接访问 `/project/:id` 时被拦截并重定向。
- 项目详情页内项目编辑、阶段库编辑和删除等管理操作只允许管理员或 PM 使用。

**Non-Goals:**
- 不新增后端权限模型。
- 不改变管理员后台 `/admin` 的权限规则。
- 不改变普通成员访问看板、列表、时间轴、成员排期、成员页的权限。

## Decisions

1. 使用现有用户信息判断项目管理权限。

   权限条件为 `user.isAdmin || user.role === 'pm'`。项目目前已有 `isAdmin` 和 `role` 字段，直接复用能保持实现简单。备选方案是新增权限表，但当前前端 mock 项目没有后端权限系统，成本过高。

2. 在顶部导航添加“项目详情”入口。

   `AppHeader.vue` 已经集中展示当前项目下的主要页面入口，把按钮放在这里比放在项目选择页更接近用户工作流。入口链接为 `/project/<currentProject.id>`，不继承 planning query，因为项目详情不是迭代视图。

3. 路由使用 meta 标记项目管理权限。

   为 `/project/:id` 增加 `requiresProjectManager` 或等价 meta，并在 router guard 中拦截无权限用户。普通成员若已有当前项目则跳转到 `/kanban/:projectId`，否则跳转 `/projects`。

4. 页面内再做控件级保护。

   即使路由已拦截，`ProjectDetailView.vue` 仍按权限隐藏或禁用“编辑项目”、阶段库编辑、新增阶段、删除项目等管理动作，避免后续入口调整时泄漏编辑能力。

## Risks / Trade-offs

- [仅前端权限无法替代后端鉴权] -> 当前项目使用 mock API，先在路由和 UI 层实现一致保护；未来接后端时需补服务端权限。
- [PM 角色名称变化会影响判断] -> 使用现有 `role === 'pm'` 常量，和系统角色定义保持一致。
- [普通成员手动输入详情 URL 体验突兀] -> 重定向到当前项目看板，让用户回到可访问的工作页面。
