## Context

当前时间轴视图将迭代范围内的每一天等同显示。实际项目中，周末和法定节假日通常不上班，但偶尔也有加班情况。需要在 Planning 数据模型中引入工作日配置，并在时间轴上可视化。

## Goals / Non-Goals

**Goals:**
- Planning 支持配置非工作日（节假日）和额外工作日（加班）
- 默认工作模式：周一~周五上班，周六日休息
- 时间轴中非工作日格子灰化显示
- 管理员可在时间轴上点击切换工作/休息

**Non-Goals:**
- 不支持不同角色有不同的工作日安排
- 不内置中国法定节假日数据库
- 不支持项目级自定义默认工作模式（固定周一~周五）

## Decisions

### 1. 数据存储在 Planning 上

**选择**：在 `Planning` 接口新增 `nonWorkdays: string[]` 和 `extraWorkdays: string[]`。

**理由**：工作日安排跟迭代强绑定（同一项目不同迭代可能有不同排休），放在迭代级别最灵活。

**替代方案**：放在 Project 级别 — 不够灵活，不同迭代的节假日不同。

### 2. 例外列表而非完整日历

**选择**：只存储"例外"日期，不存储完整的工作日列表。

**理由**：大多数日子遵循默认模式（周一~周五），只需记录例外。数据量小，维护简单。

### 3. 两个独立数组

**选择**：`nonWorkdays` 存放"应该上班但休息的日子"，`extraWorkdays` 存放"应该休息但上班的日子"。

**理由**：逻辑清晰，判断简单。不使用 `{ date, isWorkday }` 结构是因为两个数组的语义更明确。

### 4. 管理员点击切换

**选择**：在时间轴日期列头添加点击交互，管理员可切换工作/休息状态。

**理由**：最直观的操作方式——看到哪天不对，直接点一下改掉。

## Risks / Trade-offs

- **切换逻辑的边界**：点击周六 → 加入 extraWorkdays；再次点击 → 从 extraWorkdays 移除。点击周三 → 加入 nonWorkdays；再次点击 → 从 nonWorkdays 移除。逻辑清晰无歧义。
- **数据迁移**：现有 Planning 数据无新字段，需在 mock 数据中补充默认空数组。
