# 角色
	## 项目管理（PM）
	
		### 权限
			1. 新增、修改、删除、查看所有项目成员；
			2. 新增、修改、删除、查看所有项目；
			3. 新增、修改、删除、查看所有任务；
	
		### 职责
			1. 每个任务的创建者、监督人。参与任务整个过程不需要填写任务时间；
			2. 组织分配任务所需资源，包括人员、时间分配
			3. 一定参与每个任务
			
	## 策划
	
		### 权限
			1. 查看所有项目成员；
			2. 查看所有所有项目；
			3. 修改、查看任务；
			
		### 职责
			1. 每个任务的“大脑”，任务详细实现内容的创作者，所有人（PM除外）的工作的奠基者；
			2. 确定任务的具体内容和方向，需要确定策划案以及配置表
			3. 在初步开发完成后需要验收功能是否符合设计初衷
			4. 参与绝大多数任务，一些纯程序的优化除外
			
	## 美术
	
		### 权限
			1. 查看所有项目成员；
			2. 查看所有所有项目；
			3. 修改、查看任务；
		
		### 职责
			1. 在策划确定任务的具体内容和方向后，根据策划的需求设计出美术物料
			2. 不一定参与每个任务

		### 职责

			2. 不一定参与每个任务
			
	## 程序

			2. 查看所有所有项目；

		### 职责
			1. 在策划确定任务的具体内容和方向后，实现服务端逻辑、数据设计以及前端与服务端交互的端口

		- 客户端程序

			2. 查看所有所有项目；

		### 职责
			1. 在策划确定任务的具体内容和方向以及上沿美术/UI输出效果图或者物料后，实现功能客户端效果
			2. 不一定参与每个任务

# 项目
	包含多个规划（任务集）
	- 元素： 项目名
	
# 规划（任务集）
	包含多个任务
	- 元素： 规划名字、截止时间
	
# 任务
	- 元素：
		1. 参与者，在角色中，根据任务需求可添加任务参与者，其中一定参与的有PM和测试，其余可按需选择参与，默认有一个角色参与
		2. 任务时间，开始时间与结束时间。每个角色担任任务细项都按需填写时间起始，其中每个角色的细项都有制约关系：
			* 策划的策划案为完成，几乎所有其他成员的工作都无法进行；
			* UI、美术在策划奠定方向后需要输出UI或美术资源，供给程序使用；
			* 服务端程序可以在策划输出策划案之后开始着手搭建代码框架，但是部分逻辑任然需要UI输出，故而服务端程序和UI的工作可部分并行；
			* 客户端程序需要在UI输出后进行界面制作，以及前端代码编写。和服务端对接需要在服务端定好通信协议之后才能进行，故客户端程序和服务端程序的部分工作可并行；
			* 程序同学完成初版后，交由策划进行验收，确认功能是否符合预期设计；
			* 策划确认符合预期设计，和程序UI配合对功能进行完善，确认完整后交由测试对功能进行逻辑测试
			* 音效/动画特效在策划验收通过后，对功能进行润色处理，按需输出音效与动画特效
			* 音效/动画特效输出后还需要客户端程序进行接入
			* 最终成品交由测试进行稳定性测试
			* 测试期间发现的bug将再有程序进行修复，直至再无bug，该任务即算完成
		3. 任务参考，几乎每个任务都需要策划案，以及UI/美术设计图片或者链接
		4. 任务评论，每个参与者可在任务评论区进行评论，发送文档链接，图片等等
		5. 任务阶段： 立案 -> 设计 -> 初版实现 -> 初步验收 -> 终版完成 -> 最终验收 -> 完成
			
	
	- 展示形式：
		1. 普通形式，展示任务名字，所属规划，各个参与者，起始时间
		2. 时间轴形式，时间轴上用不同颜色长条展示各个角色的细项任务时间段
