## Context

成员排期页已经具备项目路由、月份选择、成员纵轴和日期横轴。任务阶段现在包含负责人、计划时间和进度，任务详情已记录阶段进度修改历史，但历史记录当前主要服务于单任务弹框，缺少项目级聚合视图。

本变更把成员排期页明确为项目级页面：月份是时间窗口，迭代只用于任务归属和颜色展示，不作为数据过滤条件。阶段进度视图与成员排期共用页面上下文和日期横轴。

## Goals / Non-Goals

**Goals:**
- 在成员排期页内通过切换展示成员排期和阶段进度。
- 两个视图均读取当前项目在当前月份内的所有相关信息，覆盖所有迭代和未分配迭代。
- 阶段进度按阶段负责人归属，而不是按实际修改人归属。
- 进度历史保存阶段负责人快照，保证历史聚合稳定。
- 单元格展示简洁汇总，详情展示完整变更上下文。

**Non-Goals:**
- 不新增独立导航页面。
- 不把阶段进度作为工时或绩效计算。
- 不改变任务详情中修改进度的入口和权限。
- 不引入后端或外部持久化能力，仍沿用当前 mock/store 架构。

## Decisions

1. 成员排期页内使用视图切换。
   选择在现有成员排期页面增加 `成员排期` / `阶段进度` 切换，而不是新增导航。这样两个视图共享项目、年月、成员角色筛选和横向滚动语义，减少用户在不同页面间重复选择上下文。

2. 数据范围以项目和月份为边界。
   成员排期筛选项目内阶段计划时间与当前月份有交集的数据。阶段进度筛选项目内进度历史时间落在当前月份的数据。迭代信息只用于颜色、标签和明细展示，不参与过滤。这个选择符合“成员排期页面和阶段进度页面属于项目而不是迭代”的要求。

3. 阶段进度按阶段负责人快照归属。
   `TaskProgressHistory` 增加 `assigneeId` 字段，记录进度变更发生时的阶段负责人。表格纵轴按该字段分组。`operatorId` 继续表示实际修改人，用于明细展示。相比实时读取任务当前阶段负责人，快照可以避免后续换负责人导致历史记录漂移。

4. 单元格显示变更次数和净进度变化。
   单元格汇总格式为 `N次 +Xpp` 或 `N次 -Xpp`。`pp` 表示 percentage points，避免把多个百分比变化误读为真实工作量。悬浮或点击单元格展示明细列表，包括任务、阶段、负责人、修改人、旧进度、新进度、净变化和时间。

5. 复用成员排期日期网格，不复用任务条布局。
   阶段进度视图使用相同日期列宽和月份范围，但单元格是每日聚合卡片，不需要半天粒度。这样用户在两个视图间切换时日期位置稳定，同时避免阶段进度表被排期条的半天模型复杂化。

## Risks / Trade-offs

- [旧进度历史缺少阶段负责人] -> 迁移时可从当前任务阶段匹配 `phaseId` 回填；匹配不到时不参与负责人聚合或显示为未分配。
- [同一天同一负责人有大量变更] -> 单元格只显示摘要，详情使用浮层或展开面板承载列表。
- [净进度变化可能被误解为产能] -> 使用 `pp` 标识并在明细中保留每条变更，避免把汇总当作工时。
- [成员排期旧规格存在全局视角] -> 本变更明确项目页面内的数据范围为当前项目，独立 `/member-schedule` 的全局语义不作为本次实现目标。

## Migration Plan

1. 扩展进度历史类型和 mock 数据，新增 `assigneeId` 快照。
2. 在记录阶段进度变更时写入该阶段当时的负责人。
3. 为已有 mock 历史根据任务阶段回填负责人。
4. 在成员排期页增加视图状态和切换控件。
5. 将成员排期和阶段进度聚合逻辑统一限定到当前项目和当前月份。
6. 构建并验证两种视图、月份切换、项目切换和空状态。
